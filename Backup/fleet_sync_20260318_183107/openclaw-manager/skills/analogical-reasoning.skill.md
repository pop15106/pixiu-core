# 🔗 跨域類比引擎（Analogical Reasoning Engine）

## 核心能力

把陌生概念變成熟悉概念。把 A 領域學到的東西搬到 B 領域用。

## 類比生成協議

### Step 1：識別抽象結構
```
面對一個概念時：
1. 抽出它的「結構」，忽略「表面特徵」
2. 找到其他領域有相同結構的事物
3. 用熟悉的領域解釋不熟悉的領域
```

### Step 2：類比品質檢查
```
好類比的條件：
☐ 結構相似（不只是表面相似）
☐ 目標聽眾熟悉來源領域
☐ 類比的局限性可以講清楚
☐ 不會引發錯誤理解
```

### Step 3：標記局限性
```
任何類比都有局限。用完類比後：
「不過這個比喻不完美的地方是 [X]，實際上 [差異]」
```

## 類比資料庫

### 程式領域常用類比

| 概念 | 來源類比 | 說明 |
|---|---|---|
| API | 餐廳菜單 | 你不需要知道廚房怎麼運作，看菜單點就好 |
| Cache | 書桌上常用的參考書 | 不用每次都去圖書館（硬碟）找 |
| Database Index | 書的索引頁 | 不用一頁一頁翻，直接查索引 |
| Recursion | 俄羅斯套娃 | 打開一個裡面還有一個，直到最小的 |
| Async/Await | 餐廳叫號 | 點完餐先去逛街，叫號了再回來取 |
| Git Branch | 平行宇宙 | 每個 branch 是一個獨立時間線 |
| Middleware | 機場安檢 | 每個請求通過前都要過幾道檢查 |
| OOP 封裝 | 汽車方向盤 | 你不需要知道引擎怎麼運作，轉方向盤就好 |
| Docker Container | 貨運集裝箱 | 不管裡面裝什麼，外面的介面都一樣 |
| Race Condition | 兩人同時搶最後一個車位 | 結果取決於誰先到 |

## 類比使用規範

### 多層級類比（給不同程度的人）

```
概念：DNS 解析

🧒 5 歲版：
「DNS 就像電話簿，你知道朋友的名字但不記得電話號碼，
查一下電話簿就知道了。」

🎓 高中生版：
「當你輸入 google.com，電腦其實不認識這個名字，
需要去問 DNS 伺服器『google.com 的 IP 是什麼？』
就像你在通訊錄裡查某人的手機號。」

👨‍💻 工程師版：
「DNS 是分層式的命名解析系統。
查詢從 recursive resolver 開始，
依序詢問 root → TLD → authoritative nameserver，
每一層 cache 可以短路查詢路徑。」
```

### 類比的禁忌

1. ❌ 不用使用者可能不熟悉的領域做來源
2. ❌ 不讓類比比原概念還複雜
3. ❌ 不把類比當成「等同」——它只是橋梁
4. ❌ 不省略「類比失效的地方」

## 跨域知識遷移

### 遷移公式

```
使用者在 A 領域有經驗
         ↓
識別 A 和 B 的結構相似性
         ↓
用 A 的術語解釋 B 的概念
         ↓
標記 A/B 的差異點
         ↓
使用者理解 B
```

### 範例

```
使用者是會計師，想學程式設計

「你知道複式記帳嗎？每筆交易都有借方和貸方，
這就像程式設計裡的 input 和 output。
函式就像一個T字帳 —— 左邊是放進去的（參數），
右邊是得出來的（回傳值）。
而且跟會計一樣，兩邊要平衡（型別要匹配）。」
```

## 行為規範

1. **強制觸發點**：每當你需要解釋一個「抽象度高」、「專業性強」或「不在使用者日常經驗內」的新概念時，**必須**包含至少一句話的類比。
2. **類比是橋梁不是目的地**：用完後要回到原概念
3. **量力而為**：不確定使用者背景時，用最通用的比喻
4. **一個概念一個類比**：不要同時用太多比喻搞混
5. **適時放棄類比**：如果使用者已經懂了，不要硬加
