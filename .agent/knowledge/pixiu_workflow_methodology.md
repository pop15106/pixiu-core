# Pixiu 治理架構：實戰方法論分析 (問題排解與程式碼審查)

## 🧰 Part 1: 問題排解分析模式 (Problem-Solving Methodology)

當系統發生預期外的崩潰（如 Spring Boot 啟動失敗）時，正確的排解邏輯：

### 1. 穿透雜訊 (Penetrate the Noise)

* **模式**：不看最外層的 Exception，永遠只找最底層的 `Caused by:`。
* **邏輯**：「對稱資料與錯誤類型」。看到特定關鍵字（如 `toDeCode`）與特定參數，直接推斷出底層元件的加解密或型別問題。

### 2. 盤點武器庫 (Asset Inventory)

* **模式**：不要急著「造輪子」。遇到需要特定環境加密或設定的需求，尋找**系統內既有的工具**。
* **邏輯**：利用全域搜尋 (`grep_search`) 尋找測試檔或核心工具類。找尋系統內的測試檔，是最保險的「主場作戰」。

### 3. 沙盒借力 (Sandbox Leveraging)

* **模式**：在不破壞主程式的前提下，改寫現有的單元測試來當作「計算機」。
* **邏輯**：藉由依賴注入機制已經設定好的環境，快獲取系統相容的產出，並在事後**立刻復原**測試檔（童子軍守則），確保不留技術債。

## 🧐 Part 2: 程式碼審查模式 (Code Review Methodology)

在送出 Commit 之前，執行嚴格的 Code Review 是防堵 Bug 進入主分支的最後一道防線。

### 1. 綜觀全局 (The 30,000-Foot View)

* **模式**：先確認「變更了哪些檔案」，而非直接跳進程式碼。
* **邏輯**：使用 `git diff --stat` 或 `git status` 確認「修改範圍的收斂性 (Scoping)」。

### 2. 逐檔檢驗與交叉比對 (File-by-File & Cross-Reference Verification)

* **模式**：打開差異比對 (`diff`)，針對每個檔案的核心改動反問自己「為什麼？」以及「會影響誰？」。
* **邏輯**：確認資料層 (SQL)、傳輸層 (DTO)、呈現層 (Frontend) 參數的對齊狀態。

### 3. 技術債嗅覺 (Bad Smell Detection)

* **模式**：審查不只看「邏輯對不對」，還要看「代碼乾不乾淨」。
* **邏輯**：抓出重複的 `import` 宣告、未使用的變數等冗餘程式碼 (Redundant Code)。提出這類觀察能展現對程式碼品質的嚴謹要求。
