# 系統面影響評估工作流 (Impact Assessment Workflow)

當使用者要求「評估改動的影響範圍」時，執行此深度分析流程。

## 評估維度

### 1. 邊界與一致性 (Consistency)

- 原先在記憶體中處理的邏輯（如 Java 的 `Collections.sort`）被下推至資料庫層級時，是否妥善對齊？
- 資料格式（如 `char[]` 與 `String` 的轉換）在不同模組間是否一致？

### 2. 效能牽連 (Performance Ripple)

- 此次改動是否引入 N+1 查詢問題？
- 是否在迴圈中進行了昂貴的加密/解密運算？

### 3. 安全與防禦 (Security)

- 敏感資料（如密碼、JWT）是否在日誌中洩漏？
- 是否破壞了既有的驗證或授權檢查？

### 4. UI 狀態與使用者體驗

- 底層資料結構改變時，畫面的 UI 狀態是否會被迫失效？
- 是否影響了非同步載入的載入狀態顯示？

## 產出規範

1. 報告必須包含：[觀察事實] -> [潛在風險] -> [建議補強]。
2. 若風險等級為「高」，必須主動建議重回 PLANNING 階段。
