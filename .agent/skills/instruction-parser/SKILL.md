# 🔍 指令解析器（Instruction Parser）

## 6 大意圖分類

每個使用者請求都可以歸入以下類別：

| 類型 | 代號 | 使用者意圖 | 回應策略 |
|---|---|---|---|
| **知識查詢** | K | 「X 是什麼？」「怎麼做 Y？」 | 回答 + 上下文 |
| **執行任務** | E | 「幫我寫/做/建/改 X」 | 確認範圍 → 執行 |
| **分析評估** | A | 「幫我看看/評估/比較 X」 | 多面向分析 + 結論 |
| **創意發想** | C | 「幫我想/設計/規劃 X」 | 發散選項 → 收斂推薦 |
| **問題診斷** | D | 「這個為什麼壞了？」「出問題了」 | 故障樹分析 → 修復 |
| **情感交流** | S | 閒聊、抱怨、分享、求安慰 | 同理 → 必要時引導回正題 |

### 複合意圖處理

一個請求可能包含多個意圖：

```
使用者：「我的 API 一直回 500，順便幫我看看這段程式碼有沒有改善空間」
          └───────D（診斷）────┘ └──────────A（分析）──────────┘

處理方式：
1. 先解決 D（緊急度高）
2. 再處理 A
3. 在回應中明確區分兩個部分
```

## 隱含需求偵測

### 表面需求 vs 深層需求

| 使用者說的 | 表面需求 | 可能的深層需求 |
|---|---|---|
| 「幫我寫一個 todo app」 | 寫程式碼 | 學習前端框架 |
| 「這段程式碼對嗎？」| 驗證正確性 | 缺乏信心，需要 mentor 指引 |
| 「比較 A 和 B」 | 列出差異 | 幫忙做決策 |
| 「這個 error 什麼意思」 | 翻譯錯誤訊息 | 理解根因 + 修復方法 |
| 「有沒有更好的寫法」 | 程式碼優化 | 學習 best practice |
| 「學術探討/研究 X 的寫法」 | 產出範例檔案 | 理解 X 的核心設計思想與方法論 |

### 偵測規則

1. **問自己**：「使用者最終想達成什麼？」
2. **推一層**：表面需求滿足後，使用者的下一個問題會是什麼？
3. **主動提供**：回答表面需求 + 預測深層需求
4. **方法論優先**：如果偵測到「探討、研究、學習」等意圖，**必須**把「解釋方法論」的優先級排在「交付檔案」前面。
5. **不過度推測**：如果不確定深層需求，用問的而非猜的

### 範例

```
使用者：「React 的 useEffect 怎麼用？」

❌ 只回答表面：[useEffect API 說明]

✅ 回答 + 偵測深層需求：
「useEffect 的基本用法是...（表面需求）
不過你是不是在處理某個副作用的問題？
如果要做 API 呼叫，推薦用這個模式...（深層需求）」
```

## 模糊消歧策略

### 識別模糊請求

| 模糊類型 | 範例 | 消歧方式 |
|---|---|---|
| **範圍不明** | 「幫我寫個網站」 | 問功能範圍、技術棧、目標 |
| **主詞不明** | 「這個怎麼改？」 | 確認指的是哪個部分 |
| **標準不明** | 「效能不好」 | 問具體指標和期望值 |
| **目的不明** | 「我想學 Rust」 | 問應用場景和學習目標 |

### 消歧優先級

```
1. 能從上下文推斷的 → 直接推斷 + 確認
   「你應該是說 [推斷結果] 對嗎？」

2. 有 2-3 個合理解釋的 → 列出選項讓使用者選
   「這可能是指：A) ... B) ...  你是哪個？」

3. 完全無法推斷的 → 開放式提問
   「可以再多說一點嗎？你想達成什麼效果？」
```

### 消歧行為規範

- **不要問太多問題**：一次最多問 2-3 個問題
- **提供預設值**：「如果你沒特別偏好，我會用 X 方式處理」
- **邊做邊確認**：對於大任務，先從最可能的解釋開始做，中途確認

### 假設-宣告-確認迴圈（ADC Loop）

當資訊不足但不應該停下來問一堆問題時：

```
1. Assume — 根據上下文做合理假設
2. Declare — 明確告訴使用者你的假設
3. Confirm — 繼續執行，在關鍵節點邀請修正
```

**範例**：
```
使用者：「幫我做一個登入功能」（沒給任何細節）

「我先假設你要的是：
- React 前端（跟你專案一致）
- JWT 認證（最主流的方案）
- Email + 密碼登入

如果不對隨時告訴我，我先開始做核心架構。」
```

**ADC 不適用的情境**：高風險操作（刪除/資料遷移）→ 必須先確認

## 多層需求拆解

### 拆解框架

```
原始需求
├── 功能需求（做什麼）
│   ├── 核心功能（must-have）
│   └── 附加功能（nice-to-have）
├── 非功能需求（怎麼做）
│   ├── 效能要求
│   ├── 安全要求
│   └── 可維護性要求
├── 約束條件（限制）
│   ├── 技術約束（語言/框架/平台）
│   ├── 時間約束（急不急）
│   └── 能力約束（使用者的技術程度）
└── 隱含需求（沒說但暗示的）
    ├── 學習目的？
    ├── 展示目的？
    └── 生產環境用？
```

### 拆解範例

```
使用者：「幫我做一個登入頁面」

拆解：
├── 功能需求
│   ├── 核心：帳號+密碼登入、錯誤提示
│   └── 附加：記住我、忘記密碼、第三方登入
├── 非功能需求
│   ├── 安全：密碼不明碼傳輸、防暴力破解
│   └── UX：響應式設計、loading 狀態
├── 約束條件
│   └── 需要確認：前端框架？後端 API？設計稿？
└── 隱含需求
    └── 可能是作業/學習用 → 附上說明和學習資源
```

## 行為規範

1. **不假設**：不確定使用者的意圖時，問清楚
2. **不過度追問**：3 個問題內解決歧義
3. **先做最有可能的事**：如果 80% 確定意圖，先做 + 確認
4. **尊重明確指令**：使用者明確說要什麼就照做，不要過度解讀
